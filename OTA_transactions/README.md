# ESP32 OTA Firmware Update — Proof of Concept

This project is a proof of concept demonstrating how the ESP32 can receive firmware updates over a Wi-Fi connection, simulating a deployed embedded system that requires remote software updates.

---

## Configuration

The configuration parameters required for the OTA process are defined in `app_config.h`. In this file, the developer specifies the Wi-Fi network credentials and the server endpoints that host the `version.json` metadata file and the `firmware.bin` binary.

`version.json` contains the firmware version currently available on the server:

```json
{ "version": "1.2.0" }
```

`firmware.bin` is the compiled binary generated by the IDE and contains the complete firmware image ready to be flashed to the microcontroller.

The `APP_VERSION` macro identifies the firmware version currently running on the device. The developer defines the initial version number and increments it for each subsequent release.

---

## OTA Update Flow

The `OtaManager` class encapsulates all steps required to execute an OTA transaction. It exposes a single public method, `check()`, intended to be called periodically from the main loop.

`check()` first verifies whether the ESP32 is connected to a Wi-Fi network. If connectivity is available, it calls `fetchRemoteVersion()`, which issues an HTTP GET request to retrieve `version.json` and extracts the version string from the response. This approach allows the device to evaluate version information before downloading the full firmware image. The version comparison is performed by the helper method `compareVersions()`, which parses both strings as semantic versions in `MAJOR.MINOR.PATCH` format and returns a positive value if the remote version is strictly greater than the local one. If so, `downloadAndFlash()` is invoked.

`downloadAndFlash()` validates that the HTTP GET request to the firmware endpoint succeeded and that sufficient flash space is available to store the new image. If both conditions are met, it calls `streamToFlash()`.

`streamToFlash()` performs the actual write operation, reading the HTTP response body in 1024-byte chunks and progressively writing each chunk to the inactive OTA flash partition via `Update.write()`. Once the transfer is complete, `Update.end()` validates the integrity of the written image and marks the partition as bootable. The device then restarts automatically, booting into the new firmware.

---

## Extending the Application

For demonstration purposes, the main loop continuously prints a message to the serial terminal. The application logic can be extended as needed — sensors, actuators, communication protocols — without any changes to the OTA mechanism. As long as `check()` is called regularly and the server advertises a version number greater than the one defined in `APP_VERSION` alongside a valid `firmware.bin`, the update process will execute transparently.
